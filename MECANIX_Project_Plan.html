1. Product Overview

MECANIX is a cloud-based, offline-first workshop management SaaS for independent automotive workshops in Lusophone markets — Angola, Mozambique, and Brazil.

The platform unifies vehicle intake, job card management, parts inventory, technician time tracking, invoicing, customer communication, and insurance claim workflows into a single affordable system accessible from any device.

Strategic Vision: MECANIX will be the first serious SaaS-native workshop management platform purpose-built for Lusophone Africa and Brazil — combining offline-first architecture, WhatsApp-native communication, a dedicated customer mobile app, and a full insurance assessor portal.

2. Architecture Overview

+-----------------------------------------------------------------------+ | MECANIX Platform | +------------------+--------------+---------------+---------------------+ | Workshop App | Technician | Customer App | Insurance Portal | | (Next.js + | App | (Expo / | (Next.js) | | Expo) | (Expo) | React Native)| | +------------------+--------------+---------------+---------------------+ | API Layer — NestJS on Fastify (TypeScript) | +-----------------------------------------------------------------------+ | Supabase: PostgreSQL (RLS) + Auth + Storage + Realtime | | PowerSync: Offline Sync (WAL -> client SQLite) | +-----------------------------------------------------------------------+ | Hosting: Vercel (web) + Railway (API) + Supabase (managed DB) | +-----------------------------------------------------------------------+ | Integrations: WhatsApp Business API | FCM | M-Pesa | Multicaixa | +-----------------------------------------------------------------------+

Data Flow

  1. Auth: User logs in → Supabase Auth issues JWT with tenant_id claim
  2. Data access: JWT drives PostgreSQL RLS → every query auto-scoped to tenant
  3. Offline: PowerSync reads PostgreSQL WAL → syncs to client SQLite → app reads/writes locally
  4. Realtime: Database changes stream via Supabase Realtime WebSockets to dashboards and mobile apps
  5. Storage: Photos/PDFs stored in Supabase Storage (S3-compatible), secured by same JWT

Why Not Clerk?

Clerk is excellent for auth + org management, but for MECANIX it creates more problems than it solves:

  • Breaks the RLS chain: Supabase Auth JWT drives RLS, PowerSync, Realtime, and Storage. Clerk would require a bridge between two auth systems.
  • Double cost: You'd pay for Clerk while Supabase Auth is already included.
  • Extra latency: Every authenticated request would pass through an additional layer.
  • The work Clerk saves is minimal: Building login screens + invite flow + role management on Supabase Auth takes ~10 hours.

Why Not Prisma?

Prisma bypasses RLS — it connects directly to PostgreSQL, not through the Supabase client. This defeats the entire multi-tenancy security model. Instead, use the Supabase JS client in NestJS, which respects RLS because it uses the user's JWT. For complex queries, use Kysely (type-safe SQL query builder).

6. Tenant Management & Billing

Tenant Management

Handled natively by the architecture — no external service needed:

  • Supabase Auth creates users and issues JWTs with tenant_id claims
  • A tenants table stores workshop profile, plan, settings, and limits
  • PostgreSQL RLS policies filter all queries by tenant_id automatically
  • Tenant onboarding: sign up → create tenant record → RLS applies → workshop operational

Subscription Billing — Hybrid Approach

No single billing provider covers all 3 markets. The approach is build the plan engine, outsource payment collection where possible.

Custom BillingModule (NestJS)

A lightweight module that handles:

  • Plan logic: 5 tiers with clear limits (max job cards, max users, features)
  • Trial management: 14-day free trial → prompt to subscribe
  • Subscription invoicing: Monthly invoice PDF generation (reuses existing engine)
  • Limit enforcement: NestJS guard checks plan limits before creating resources
  • Grace period: 7 days past due → suspend → 30 days → archive
  • WhatsApp reminders: Automated payment reminders at 7, 3, and 1 day before due

7. Key Architectural Decisions

1. Supabase is the backbone — not just a database. Auth, storage, realtime, and RLS all flow from a single JWT. Do not break this chain with external auth (Clerk) or external ORM (Prisma).

2. No Prisma — use the Supabase JS client, which respects RLS. Prisma bypasses RLS and defeats the entire security model. For complex queries, use Kysely or raw SQL via Supabase's rpc() function.

3. No Clerk — paying for two auth systems while maintaining a bridge adds cost and complexity. Supabase Auth does everything needed and integrates natively with RLS, PowerSync, Realtime, and Storage.

4. Billing is hybrid — build a simple plan/limits engine. Angola and Mozambique run on bank transfers and manual confirmation. Automate with Stripe only when entering Brazil.

5. Offline-first is non-negotiable — PowerSync from day 1. Every mobile feature must work without internet. This is a market requirement in Angola and Mozambique.

6. Money as integers — all amounts stored in cents/centavos. No floating point. No exceptions. Retrofitting this is extremely painful.

7. i18n from day 1 — no hardcoded strings, ever. pt-PT, pt-BR, en. All formatting (dates, numbers, currency) locale-aware via the Intl API.

9. Sprint Plan — 20 Weeks

Team: 1-2 developers + Claude Code as force multiplier
Sprint length: 2 weeks
Total sprints: 10 (Sprint 0-9)

Sprint 0 — Stack Ramp-Up

Weeks 1-2  |  ~30 hours

Supabase quickstart, NestJS scaffold, Expo app, PowerSync offline test, Next.js setup, deploy everything (Railway + Vercel), GitHub Actions CI, i18n setup. Output: Throwaway proof-of-stack where all pieces connect.

Sprint 1 — Foundation & Tenant System

Weeks 3-4  |  ~44 hours

Production monorepo, database schema (tenants, users, customers, vehicles) with RLS, TenantGuard, auth flow, tenant onboarding, login/register screens (mobile + web), PowerSync production sync rules, i18n configuration, deploy pipeline.

Sprint 2 — Customer & Vehicle Management

Weeks 5-6  |  ~44 hours

Full CRUD for customers and vehicles, search (name/phone/plate/VIN), vehicle service history, photo uploads, customer list/detail pages (web), mobile customer search + vehicle registration with camera, offline create/sync verification.

Sprint 3 — Job Card System (Core Product)

Weeks 7-8  |  ~61 hours (heaviest sprint)

Job card CRUD with auto-numbering, 9-status workflow engine with validation, labour lines, parts lines, job card settings (insurance toggle, digital auth, parts issuing mode, taxable toggle), technician profiles, job card list/detail pages (web + mobile), offline job cards.

Sprint 4 — Technician Time Logging

Weeks 9-10  |  ~47 hours

Dedicated technician Expo app: one-tap start/stop timers, clock in/out, job interaction (notes, photos, flags), productivity dashboard, live floor view (manager web), daily timesheet report, offline time tracking, UX review for gloved hands.

Sprint 5 — Parts, Inventory & Suppliers

Weeks 11-12  |  ~52 hours

Parts catalogue, stock management with low-stock alerts, parts issuing workflow (auto vs manual), service groups, inventory adjustments, vendor management, purchase orders, bills, vendor credits, expense tracking with receipt uploads.

Sprint 6 — Invoicing & Payments

Weeks 13-14  |  ~50 hours

Auto-generate invoices from job cards, configurable tax engine (IVA 14%/17%), PDF generation with branding, payment recording (cash/transfer/card/M-Pesa/PIX), partial payments, split billing (customer + insurer), credit notes, financial dashboard widgets, accounts receivable aging.

Sprint 7 — WhatsApp + Customer App

Weeks 15-16  |  ~47 hours

WhatsApp Business API integration, message templates for all triggers, notification on status changes, SMS fallback, Customer App: phone OTP onboarding, my vehicles, live job tracking with push notifications, digital quote approval, invoice view, service history, Firebase Cloud Messaging setup.

Sprint 8 — Insurance Module

Weeks 17-18  |  ~54 hours

Insurance claim 7-stage workflow, structured estimate submission, assessor web portal (claims queue, line-by-line estimate review, rate card comparison), mandatory GPS-tagged damage photos, InsuranceGuard (cross-tenant claim access), fraud detection (duplicate claims, anomaly flagging), repair monitoring, workshop-side insurance job card enhancements.

Sprint 9 — Dashboard, Reports & Beta Launch

Weeks 19-20  |  ~62 hours

Owner dashboard (Kanban jobs, revenue, receivables/payables, income vs expense), 11 report endpoints with PDF export, service reminder engine, workshop settings (bays, tax, branding), offline sync hardening, performance testing (50 concurrent users), i18n review, bug bash, beta onboarding (3-5 workshops), Sentry monitoring, quick-start documentation.

11. Risk Flags & Recommendations

RiskImpactMitigation
PowerSync is the biggest technical risk High Invest Sprint 0 in a proof-of-concept. Test offline read/write/sync cycle before writing production code.
WhatsApp Business API approval takes 2-4 weeks Medium Apply for Meta Business account immediately, even before coding starts. Use 360dialog BSP if direct access is slow.
Workshop adoption resistance High Simple onboarding (<30 min), free setup for first 20 customers, champion program.
Connectivity causing data loss (AO/MZ) High Offline-first architecture with PowerSync. Conflict resolution: server-authoritative for status, merge for time entries, append-only for photos.
Sprint 3 (Job Cards) may overrun Medium This is the heaviest sprint (~61h). Allow 1-2 days spillover into Sprint 4. Job card system must be solid before building on top.
No billing provider covers Angola/Mozambique Medium Hybrid approach: semi-manual billing (invoice + bank transfer + admin confirmation) for AO/MZ, Stripe for Brazil.
Insurance company reluctance to adopt High Start with 1-2 insurer partnerships. Prove ROI with pilot data. Offer free onboarding.

Scope deferral priority — if any sprint runs over, defer these first (without impacting sellability): vendor credits, service groups, fraud detection, customer app payments, expense tracking. Never defer: job card lifecycle, offline sync, WhatsApp notifications, invoice generation, basic insurance workflow.
